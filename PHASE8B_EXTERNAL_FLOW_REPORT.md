# Phase 8B External Flow Hardening Report

## Scope and Goal
Phase 8B focuses on hardening the local inbox external simulation as a reliable Manager-owned entry:

`inbox/ -> processing/ -> adapter normalize/validate -> Manager ingest -> delegate_task -> processed/ or rejected/`

Constraints honored:
- No real Trello API integration
- No real Git commit/push automation from pipeline
- No Phase 6 core runtime edits (`delegate_task.py` / `worker_runtime.py` unchanged)
- No DB/message queue/UI expansion

## Input Validation Rules (Adapter)
Implemented in `adapters/local_inbox_adapter.py`:
- `title` required, non-empty (practical min length 3)
- `description` required, non-empty (practical min length 10)
- `labels` optional, but if provided:
  - must be array of strings
  - each label must match `^[a-z0-9][a-z0-9_.:-]{0,31}$`
- `metadata` optional, but if provided must be object
- `input` optional, but if provided must be object
- unsupported `request_type` is rejected

Invalid input never enters dispatch:
- directly rejected
- `.result.json` persisted with reason

## Task Legality Validation
After adapter normalization:
- both automation and critic task payloads are validated with `validate_task`
- if either fails:
  - do not call `delegate_task`
  - move to `rejected/`
  - write `.result.json` with validation errors

## Dedupe / Idempotency Rules
Implemented in `skills/ingest_tasks.py`:
- Dedupe keys are generated by adapter:
  - priority key: `origin:<origin_id>` when origin_id is provided
  - fallback key: `hash:<sha256(canonical_payload)>`
- Seen keys are loaded from historical `processed/*.result.json` and `rejected/*.result.json`
- Any inbox item with seen dedupe key is not executed:
  - moved to `rejected/`
  - decision marked `duplicate_skipped`
  - explicit duplicate key reason recorded

## Processing State Flow Hardening
Flow now:
- `inbox/*.json` is first claimed into `processing/`
- processing file is then finalized into:
  - `processed/` on success decision
  - `rejected/` on validation/duplicate/execution rejection
- crash recovery behavior:
  - pre-existing `processing/*.json` are recovered and handled on next run
- every finalized file has `.result.json` sidecar in `processed/` or `rejected/`

## Critic Structured Verdict Integration
- Critic verdict normalized to one of:
  - `pass`
  - `fail`
  - `needs_revision`
- Verdict extraction precedence:
  1. `critic_result.metadata.verdict` if valid
  2. `critic_result.summary` keyword
  3. review artifact content (`verdict: ...` or keyword)
  4. fallback (`needs_revision`, or `fail` if critic status failed)
- Manager ingest remains decision owner:
  - `verdict=pass` -> `processed`
  - otherwise -> `rejected`
- Critic does not dispatch tasks and is not a second scheduler
